Scaling Latent Reasoning via Looped Language Models
Rui-Jie Zhu, Zixuan Wang, Kai Hua ほか32名（Yoshua Bengio 含む） arXiv cs.CL ・ 2025年10月投稿（2026年7月改訂） 原文 →

1なぜ今「考える場所」が問われているのか

今のLLMが「賢く」見えるとき、その多くはChain-of-Thought (CoT)途中の思考過程を文章として書き出しながら答えを導く手法のおかげです。ただしCoTには根本的な制約があります。モデルは事前学習では「次の単語を当てる」ことだけを学び、「推論する」という能力は事後の強化学習やファインチューニングで後付けされます。つまり、事前学習という一番データ量の多いフェーズで、モデルの「考える力」そのものはあまり鍛えられていないのです。今日紹介する論文は、この前提そのものを疑い、「推論を事前学習の段階からモデルの構造に埋め込んでしまおう」という大胆な提案をしています。

2背景: CoTは「声に出して考える」方式

CoTは例えるなら、頭の中で考えず、常に独り言をブツブツ喋りながら思考する人のようなものです。喋った言葉(トークン)がそのまま思考過程になるので、長く考えたいほど大量のトークンを生成しなければならず、計算コストも待ち時間も増えます。また、喋りながら考えている内容(CoTの文章)が、実際に脳内で起きている計算過程を正確に表しているとは限らない、という指摘も近年よくされています。もし「声に出さず頭の中だけで考える」computational な仕組みをモデルに持たせられたら、もっと効率よく、かつ実態に忠実な推論ができるかもしれません。

3Ouroの全体像: Looped Language Model

著者らが提案するのが Ouro（自分の尾を食べる蛇 Ouroboros にちなむ名前）と呼ばれる Looped Language Model (LoopLM)同じ計算ブロックを何度も繰り返し通すことで、出力せずに内部で計算を深める言語モデル です。通常のTransformerは層を1回ずつ順番に通過するだけですが、Ouroは同じブロックを 潜在空間人間には直接読めない、モデル内部の数値表現(隠れ状態)の空間 の中で何度もループさせ、次のトークンを出す前に「頭の中で考え直す」ステップを挟みます。

# LoopLMは同じブロックを繰り返し通し、内部で『考え直す』

3z = shared_block(z)①
4stop_prob = halting_head(z)②
5if should_stop(stop_prob):
6break③
# 十分『考えた』ところで初めてトークンを出力
7next_token = output_head(z)

① 潜在空間での反復計算

ループのたびにモデルは隠れ状態zを少しずつ更新していきます。これは下書きを何度も推敲するようなイメージです。1回目の下書き（1回目のループ）よりも、5回書き直した下書き（5回ループ後）の方が、より練られた「考え」になっている、というわけです。しかもこの推敲はすべて潜在空間人間には直接読めない、モデル内部の数値表現(隠れ状態)の空間の中で完結するため、CoTのように大量の単語を生成する必要がありません。

② どれだけ考えるかを自分で学習する

ずっとループし続ければ精度は上がるかもしれませんが、計算コストがかさみます。そこでOuroは、各トークンに対して「あと何回ループすべきか」という深さ配分自体を学習します。論文ではこれをエントロピー正則化出力の確信度にばらつきを持たせすぎないよう罰則を与える学習テクニックを使った目的関数で実現しており、簡単な部分は少ないループで、難しい部分にはより多くのループを割り当てることを狙っています。これは著者らが7.7兆トークンという大規模なデータでこの仕組みを事前学習の段階から鍛え上げた、という点が特に野心的です。

4新規性: なぜこの結果が重要なのか

著者らは対照実験を通じて、Ouroの強さの正体を掘り下げています。もし単に「モデルが多くの知識を記憶している」だけなら、それはパラメータ数を増やすのと本質的に同じ話です。しかし論文が示したのは、Ouroの優位性は知識の量ではなく、持っている知識をどう組み合わせ、どう推論に使うかという「知識操作能力」に由来するという点でした。さらに興味深いのは、LoopLMが内部で行っている推論の跡（潜在空間での計算過程）が、CoTの文章よりも最終的な出力と整合的である、という指摘です。これは「モデルが本当は何を考えて答えを出したのか」を理解する上でも意味のある発見です。

5意義と限界

限界1

検証されているのは1.4B・2.6Bという比較的小さいモデルであり、「既存の最大12B級モデルに匹敵する」のはあくまでこの規模での話。LoopLM自体をさらに大規模化した場合にも同じ優位性が保たれるかはまだ実証されていません。

限界2

同じブロックを何度もループさせる設計上、1トークンあたりの計算回数は通常のTransformerより増える可能性があり、推論時の速度・コストとのトレードオフを常に意識する必要があります。

限界3

「知識操作能力が向上した」という説明は対照実験に基づく解釈であり、なぜ潜在空間でのループがその能力を引き出すのかという、より深いメカニズムの解明は今後の課題として残されています。

💡 R&Dでよく出る考え方
「推論をどこで行わせるか」は、CoTのように出力トークンとして表に出すか、Ouroのように内部の潜在空間に隠すか、という設計上の大きな分岐点です。新人研究者のうちは、ある手法を見たときに「これは計算をどこに、どれだけ割り当てているのか」という視点で読み解く癖をつけると、アーキテクチャ間の違いがぐっと理解しやすくなります。
