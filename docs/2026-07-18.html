Per-Token Fixed-Point Convergence in Depth-Recurrent Transformers
著者情報は元論文(arXiv)を参照 arXiv cs.AI ・ 2026年7月 原文 →

1なぜ「深さ方向にループするTransformer」が気になるのか

大規模言語モデルに難しい問題を解かせるとき、いま最も普及している方法はChain-of-Thought (CoT)途中の思考過程をトークンとして書き出しながら推論する手法です。考える過程をすべて言葉にして書き出すぶん、賢くなる代わりにトークン数が増え、時間もコストもかさみます。推論コストの高さは、いまR&Dの現場でも頻繁に議論になる話題です。

もう一つの道として研究されているのが、トークンを増やす代わりに、同じ層を繰り返し適用して「頭の中で」考える深さを増やすというアプローチです。これを深さ再帰Transformer(depth-recurrent transformer)と呼びます。今日の論文は、この「頭の中でループする」モデルが実際に何を計算しているのかを、初めてトークン単位で直接測定した研究です。

2深さ再帰Transformerのしくみ

深さ再帰Transformerは、weight-tied複数のステップで同じ重み(パラメータ)を使い回すことな「コア」ブロックを1つだけ持ち、それを何回も繰り返し適用します。普通のTransformerが「層1→層2→層3→…」と別々の重みを積み重ねるのに対し、こちらは「同じ層を1周、2周、3周…」とループさせるイメージです。

先行研究では、学習時にループ回数をランダムに変えて訓練しておくと、1つのモデルをさまざまな推論の深さ(ループ回数)で使い回せることが分かっていました。しかし「ループを重ねるたびにモデルの中で何が起きているか」「どのくらいループすれば十分なのか」は、これまできちんと測られていませんでした。

3測定方法: ループのたびに出力がどう変わるかを見る

著者らは、FineWeb-Eduで学習した135M-classモデルパラメータ数が約1億3500万程度の小規模な言語モデルを使い、ループを1回増やすごとに次トークンの予測分布がどれだけ変わるかを、KLダイバージェンス2つの確率分布がどれくらい異なるかを測る指標。0に近いほど「ほぼ同じ」で測定しました。

ループを重ねるほど出力の変化は小さくなり、やがてほとんど動かなくなる ― 各トークンが独自の不動点に収束していく
# 1トークンぶんの処理: 出力が落ち着くまでループする
1state = init_state(token)
2prev_output = None
# 同じ重みのコアブロックを繰り返し適用
3for loop in range(MAX_LOOPS):
4state = core(state)
5output = readout(state)
6if prev_output and KL(output, prev_output) < THRESHOLD:
7break①
8prev_output = output
training-freeな早期打ち切りルール(論文中の停止規則を簡略化して図示)

発見① モデルは本当に「不動点」に収束していた

2回目のループでの出力変化(KLダイバージェンス)は0.39もありましたが、16回目には0.0000085まで縮みました。ループのたびに前回との差がどんどん小さくなっていき、不動点同じ処理を繰り返しても出力がほぼ変化しなくなる状態に落ち着いていく様子が、そのまま観測されたのです(①のbreakが発生するタイミングに相当します)。

発見② 収束の速さはトークンによってバラバラ

面白いのはここからです。全トークンが同じ速さで収束するわけではありませんでした。中央値では6ループ目には収束するのに、およそ10%のトークンは学習時の平均深さである8ループ目になってもまだ出力が動き続けていました。しかも収束の速さにはきれいな傾向があり、空白のようなほぼ自明なトークンは浅いループで収束し、意味の重い内容語ほど深くまでループを必要としました。

たとえるなら、文章を読むときに私たちが「ここは軽く目を通せばいい部分」と「ここはじっくり考えないといけない部分」を無意識に切り替えているのに近い感覚です。深さ再帰Transformerは、この「軽く/じっくり」の切り替えをループ回数という形で自然に獲得していたことになります。

さらに著者らは、この打ち切りが機能する背景として、モデルの弾力性1回だけ学習したチェックポイントを、推論時に異なるループ回数で使い回せる性質を確認しています。検証損失は1ループの3.80から8ループの3.20まで単調に下がり、そこから32ループまで悪化せず安定していました。つまり「ループ回数を減らしても壊れない」土台があってこそ、トークンごとに打ち切るという工夫が安全に機能するわけです。

発見③「予測する」より「読み取る」ほうが強かった

「あとどれくらいループが必要か」を外から学習して当てさせるルーター入力に応じてどの経路・深さを使うか選ぶ小さな判定器も試されましたが、うまくいきませんでした。学習したルーターはほぼフル深さを要求してしまい、計算削減の効果がほとんど得られなかったのです。一方で、単純に「前回との出力差が小さくなったら打ち切る」だけのtraining-freeなルールは、深さ8固定と同じ品質を保ったまま、平均ループ数を4.94回まで減らすことに成功しました。

4意義と限界

この結果は、テスト時に計算量を賢く配分するアダプティブ計算入力の難しさに応じて計算量を変える仕組みの設計に直接効いてきます。追加の学習なしに「トークンごとに必要なだけループする」だけで、平均計算量を4割近く減らせるというのは、推論コストが問題になりがちな大規模モデル運用にとって実用的な示唆です。

モデルの中身をブラックボックスのまま扱うのではなく、「実際に何が起きているか」を直接測ってから対策を考える、というのはこの分野で繰り返し効く定石です。学習で解決しようとする前に、まず観測できないか考えてみましょう。
限界1

実験は135M程度の1つのスケール・1つの乱数シードでのみ確認されており、より大きなモデルでも同じ傾向が成り立つかは未検証です。

限界2

平均ループ数の削減はFLOPの目安として報告されているだけで、実際の実行時間短縮を保証する厳密な比較(FLOPを揃えた公平な比較)は行われていません。

限界3

研究全体が1台のRTX 4090・約100GPU時間という比較的小規模な計算資源で行われており、追試や拡張の余地が大きく残されています。
